## Summary

The type-at-cursor / paste behavior is inconsistent depending on the focused application. Currently paste works in the terminal but fails in Kate. Previously the opposite was true (worked in Kate, failed in terminal). Behavior is not reliable across apps.

## Steps to reproduce

1. Launch Multimodal Voice Typer.
2. Focus a terminal window (e.g., Konsole), dictate — paste succeeds.
3. Focus Kate, dictate — paste does not occur at the cursor.
4. Previous builds exhibited the inverse: worked in Kate, failed in terminal.

## Expected behavior

Dictated and cleaned-up text should be reliably pasted at the cursor in whichever application currently has focus, across terminals, text editors (Kate), browsers, IDEs, etc.

## Actual behavior

Paste-at-cursor works intermittently and its success depends on the target application. At least one common target (Kate or terminal) is broken at any given time.

### Root cause

`_inject_text` in `app/src/main.py` always sent `Ctrl+Shift+V` via ydotool.
That shortcut is correct for terminals (Konsole etc., where plain `Ctrl+V`
sends a literal control character), but in most other apps — Kate, browsers,
IDEs — `Ctrl+Shift+V` is bound to "paste special / clipboard history" or is
unbound, so the paste silently no-ops. Commit 3904229 had swapped the other
way; either single-shortcut choice leaves one class of app broken.

### Fix

- Added `_detect_active_window_class()` that asks KWin (via `kdotool
  getactivewindow` + `getwindowclassname`) for the focused window's class.
  `kdotool` talks to KWin over D-Bus, so it works for both Wayland-native
  and XWayland windows.
- Added `_paste_shortcut_for()` which returns `ctrl+shift+v` for a
  hard-coded set of terminal classes (konsole, yakuake, alacritty, kitty,
  wezterm, gnome-terminal, xterm, tilix, terminator, ptyxis, foot, ghostty,
  etc.) and `ctrl+v` for everything else.
- `_inject_text()` now copies to clipboard, detects the active window
  class, picks the right shortcut, and sends it via `ydotool key`.
  Clipboard restore logic and error handling are unchanged.
- Falls back to `ctrl+v` if `kdotool` is missing or the class can't be
  read — safe default that covers most non-terminal apps.
- `VOICE_TYPER_DEBUG=1` env var enables a stderr trace of the decision.

### Files changed

- `app/src/main.py` — `_inject_text`, plus two new helpers and a
  `_TERMINAL_CLASSES` set.

### Verification

- Code-level: reviewed control flow, confirmed `kdotool` returns class

- Build: `./build.sh --dev` produced and installed
